GitHub 官方维护的安全查询引擎。通过构建代码关系型数据库，精确定义跨文件的 isSource() 和 isSink()，是目前质量最高但也最重的基座。

github/codeql (CWE rules)

基于直观 YAML 声明的快速分析器。最大优势是无需编译即可运行，非常适合提取常见的危险函数误用组合或数组越界特征。

returntocorp/semgrep-rules

2

第二层：上下文组装与程序切片 (Program Slicing)

从庞大的代码库中抽取局部逻辑，消除噪声，为大模型提供精准“子弹”。

逆向切片 (Backward Slicing)

从可疑参数（如 memcpy 的 size）出发，逆向追踪控制流，让 LLM 确认该参数在进入危险 Sink 前是否经过安全校验清洗（Sanitization）。

跨层级调用图 (Call Graph)

基于 Tree-sitter 或 LSP(Clangd) 向上追溯 1 层 Caller，判断入参是否源自不可信的用户外部网络请求；向下追溯 Callee，防止 LLM 产生误报幻觉。

结构体宏定义 (Definitions)

大模型对复杂的 C/C++ 内存布局缺乏先天感知。必须附带提取出相关 struct 的声明，以便模型能准确计算指针偏移与缓冲区容量。

CONSTRAINT (防御幻觉): 你只能依赖我提供的片段进行推演。绝对不要假设在提供的代码之外存在任何未知的过滤函数或缓冲区长度限制！

CoT (思维链):
1. 追踪变量 Source 是否可控。
2. 检查上下文中是否存在清洗/校验机制。
3. 结合内存结构体大小计算是否越界。

OUTPUT: 仅输出包含 `is_vulnerable` (bool) 和 `reason` 的标准 JSON。

工业落地实施建议 (Implementation Paths)

根据研发环境资源，推荐以下两条工程实现路径：

极速免编译路线 (MVP推荐)

  • 场景：IDE 插件或 Pre-commit 检查
  • L1 工具：Semgrep + Flawfinder XML
  • L2 切片：Tree-sitter (Python binding)
  • 优点：无需构建脚本，兼容残缺工程，一周可完成原型。

高精度 CI/CD 路线

  • 场景：服务端全量核心资产扫描
  • L1 工具：CodeQL 或 Clang-Tidy
  • L2 切片：Joern CPG 深度跨文件
  • 优点：分析极其精确，基本消除由于缺乏编译期类型知识导致的假阳性。
